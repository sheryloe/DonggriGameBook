# PRN · Story 01 여의도 입성

## 1. 기준 자료

- 세계관 기준: `NAKWON: LAST PARADISE` 공식 티저 페이지
- 내부 확장 기준: [`docs/world/yeouido-world-bible.md`](/mnt/d/SYSRND/Codex/DonggriGamebook/docs/world/yeouido-world-bible.md)
- 구현 형태: `Vite + React + TypeScript + React Router`
- 배포 기준: `Cloudflare Pages`

공식 설정에서 차용한 축은 다음 세 가지다.

1. 서울, 특히 여의도 안전지대가 주요 거점이다.
2. 낮에는 거점 생활, 밤에는 도시 탐사와 회수 임무가 반복된다.
3. 감염자뿐 아니라 다른 생존자와의 접촉도 위협이 된다.

Story 01에서는 이를 더 좁혀 아래처럼 고정한다.

- 여의도는 `운영위원회`와 산하 실무조직이 굴리는 생존권이다.
- 신규 입성자는 `기록국(RB)`, `배급국(RD)`, `현장회수반(FR)`, `출입통제반(GC)`의 절차를 차례로 통과한다.
- 엔딩 차이는 감정적 승패보다 `행정 상태`의 차이로 남긴다.
- 화면은 `Title Screen -> Onboarding / Processing -> Journal / Inventory Shell -> Failure / Ending Screen` 순서의 Stitch형 문법을 따른다.
- Story 01은 짧은 데모가 아니라 `10분 이상` 체감되는 한 편의 에피소드로 설계한다.
- 기본 박자는 `등록 -> 배정 -> 공개 압박 -> 브리핑 -> 병목 통과 -> 외곽 진입 -> 수색 -> 회피 -> 접촉 -> 귀환 판정 -> 엔딩`이다.
- 엔딩은 종료 화면이 아니라 `판정 문서 + 인장 + 다음 호출`이 붙은 절차로 다룬다.

## 2. 경험 목표

- 사용자가 설명 없이도 첫 장면부터 마지막 엔딩까지 진행할 수 있어야 한다.
- 시작은 곧바로 본편이 아니라 `이름 입력 -> 임시 코드 발급 -> 손등 스탬프 -> 열감지 스캔 -> 규칙 방송`의 처리 흐름으로 열린다.
- 텍스트는 건조한 생존 로그 톤을 유지하되, 장면별 경보판 UI로 긴장을 보강한다.
- 첫 화만으로 `여의도의 질서`, `야간 탐사의 위험`, `생존자 간 불신`이 전달되어야 한다.
- 첫 화만으로 `거주 신분`, `배급-공제 경제`, `검문 정치`, `비공식 거래`의 존재가 느껴져야 한다.
- 시스템은 `보급`과 `소음` 두 값만 사용해 진입 장벽을 낮춘다.
- 본편 화면은 `journal / inventory shell`처럼 보여야 한다. 즉, 이야기와 물자, 기록과 상태가 한 화면의 장치처럼 이어져야 한다.
- 플레이어가 기억해야 하는 것은 단일 선택보다 `어느 줄에 섰는가`, `누구와 접촉했는가`, `어떤 문구가 남았는가`다.
- 각 엔딩은 다음 회차를 부르는 재진입 문구를 남겨야 한다. 성공은 배치의 시작, 부채는 감시의 시작, 격리는 보류의 시작으로 읽히게 한다.

## 3. 런칭 흐름

### 3.1 Title Screen

- 게임은 타이틀 화면에서 시작한다.
- 타이틀 화면은 분위기, 세계관, 시작 버튼, 불러오기, 옵션, 실패 기록 접근을 보여준다.
- 첫 화면은 아름다운 메뉴가 아니라 `입장 전 문서`처럼 보이게 한다.

### 3.2 Onboarding / Processing

1. 이름 입력
2. 임시 코드 발급
3. 손등 스탬프
4. 열감지 스캔
5. 규칙 방송
6. 임시 체류자 등록

이 구간은 Story 01의 실제 첫 노드가 아니라, Story 01로 진입하기 전의 `처리 장면`이다. 사용자는 여기서 이미 안전지대의 질서에 편입되며, `사람`이 아니라 `등록 대상`으로 다뤄진다.

### 3.3 Journal / Inventory Shell

- 본편 플레이 중에는 장면 카드, 상태 HUD, 이동 기록, 간단한 물자 슬롯이 함께 보여야 한다.
- 인벤토리는 RPG처럼 복잡하지 않고 `기록 가능한 것만 남는 간단한 패널` 정도로 제한한다.
- 시각 구조는 `좌측 기록 / 중앙 장면 / 우측 상태` 또는 모바일에서는 `상단 상태 / 본문 / 하단 선택지`로 축약한다.

### 3.4 Failure Screen

- 실패 화면은 화려한 게임오버가 아니라 `검문 결과`와 `격리 또는 공제 판정`을 보여준다.
- 실패는 죽음만이 아니라 `배급 제외`, `조건부 통과`, `격리 대기`까지 포함한다.
- 실패 화면의 핵심 문장은 감정적 위로보다 행정 문구여야 한다.
- 실패 화면은 종료가 아니라 `기록 보존`과 `재진입 조건`을 함께 보여줘야 한다.
- 성공 엔딩도 실패 엔딩과 같은 문법을 공유해야 한다. 승인, 공제, 보류를 모두 `판정 카드`로 묶고, 플레이어가 다음 행동을 바로 이해하게 만든다.

## 4. 라우팅 및 상태 계약

- 기본 경로: `/story/1/:nodeId`
- 시작 노드: `intake`
- 직접 URL 진입 허용: `night-gate`, `return`, `ending-clear` 같은 하위 장면도 직접 렌더링 가능해야 한다.
- 상태값:
  - `supplies`: 회수 실적과 선택지 여유를 나타냄
  - `noise`: 감염자 및 검문 리스크를 나타냄
- 제한 사항:
  - v1에는 인벤토리, 세이브 슬롯, 서버 저장을 넣지 않는다.
  - 상태값은 0~9 범위에서만 움직인다.

### 세계관 번역 규칙

- `supplies`는 식량만이 아니라 배터리, 약품, 배급 우선권, 회수 실적을 묶은 값이다.
- `noise`는 감염자 반응뿐 아니라 검문 기록과 접촉 흔적을 포함한 사회적 위험도다.
- Story 01에서 자주 나오는 코드 체계는 아래를 따른다.
  - `B-17B`: 숙영 코드
  - `W-03`: 서문 차단 슬롯
  - `N-01`: 북문 귀환 검문대
  - `Q-2`: 격리 대기동
  - `A-DEBT`: 조건부 통과 태그

## 5. 노드 설계

| Node ID | 장면 | 목적 | 대표 분기 |
| --- | --- | --- | --- |
| `intake` | 입성 심사 | 세계관 도입, 신규 입성자 포지션 부여 | 솔직한 답변 / 상처 은폐 |
| `barracks` | 임시 배정 | 여의도 내부 질서 체험 | 정량 배급 / 추가 챙기기 |
| `briefing` | 첫 지시 | 야간 회수 루프 정의 | 우회로 / 빠른 루트 |
| `shutter` | 방화셔터 병목 | 공개 압박과 줄 서기 정치 | 대기 / 선점 |
| `night-gate` | 야간 진입 | 안전지대 밖으로 전환 | 서쪽 통로 / 정문 잔해 |
| `arcade` | 자원 수색 | 회수품 확보 | 약국 / 자판기 |
| `collapse` | 감염자 회피 | 소음 리스크 강조 | 기어가기 / 유도하기 |
| `contact` | 생존자 접촉 | 거래와 배신 가능성 제시 | 교환 / 절도 / 정보 매매 |
| `return` | 귀환 판단 | 엔딩 갈림길 | 표준 절차 / 조건부 통과 / 격리동 우회 |
| `ending-clear` | 정식 임무 자격 확보 | 성공 엔딩 | 다시 시작 |
| `ending-debt` | 부채 태그 부착 | 조건부 생존 엔딩 | 다시 시작 |
| `ending-quarantine` | 격리 또는 임무 실패 | 실패 엔딩 | 다시 시작 |

Story 01의 본문 목표는 `12~14개의 박자`를 느끼게 하는 것이다. 현재 표의 핵심 노드 외에, 브리핑과 귀환 사이의 `공개 낙인`, `병목 대기`, `소문 확인`, `회수 후 재검문` 성격의 장면이 중간 박자로 작동해야 한다.

## 6. 시각 설계

- 공식 이미지 직접 삽입 금지
- 구현 원칙:
  - 타이틀 화면 1개
  - 처리/등록 화면 1개
  - 장면별 인포그래픽 패널 1개
  - 상태 HUD 1개
  - 로그 패널 1개
  - 실패 화면 1개
- 색 문법:
  - 안전지대 내부: 청록/황색 규율
  - 외부 탐사: 청색/분홍 잔광
  - 군집 위협: 적색 파형
  - 귀환 판정: 녹색 스캔
  - 부채/감시: 호박색
- 마모 규칙:
  - 새 장비가 아니라 `재활용 표지판`, `번진 도장`, `겹겹이 붙은 경고 테이프`, `물 얼룩`, `수기 수정 흔적`이 보여야 한다.
- 산출물:
  - 실행용 자산 매니페스트: [`src/data/story1Assets.ts`](/mnt/d/SYSRND/Codex/DonggriGamebook/src/data/story1Assets.ts)
  - 생성 프롬프트 문서: [`docs/assets/story-01-prompts.md`](/mnt/d/SYSRND/Codex/DonggriGamebook/docs/assets/story-01-prompts.md)

## 7. 구현 단계

### STEP A

- Vite 프로젝트 생성
- BrowserRouter 구성
- Cloudflare Pages용 `_redirects` 추가
- Title Screen과 Onboarding / Processing Shell 구성

### STEP B

- `StoryNode`, `StoryChoice`, `StoryState`, `EndingSummary`, `AssetSpec` 타입 정의
- 스토리 데이터와 부트스트랩 상태 정의
- 이름 입력 및 임시 코드 발급 상태를 스토리 시작 메모리와 연결
- 초기 구현 후에는 `중간 사건 노드`를 2~4개 더 추가해 총 플레이 시간을 10분 이상으로 맞춘다.
- 선택지는 단순 분기보다 `소음 관리`, `체류권 관리`, `목격담 관리`를 서로 다르게 만드는 역할을 해야 한다.

### STEP C

- 장면 카드
- 상태 HUD
- 선택지 카드
- 이동 로그
- 엔딩 리포트
- journal / inventory shell 레이아웃
- 엔딩 리포트는 `승인`, `부채`, `격리`를 각각 다른 문장 구조와 시각 톤으로 보여준다.
- `ending-clear`는 배치 통보, `ending-debt`는 공제 통보, `ending-quarantine`은 보류 통보로 읽혀야 한다.

### STEP D

- 장면별 CSS 인포그래픽 구현
- 프롬프트 문서 분리
- 각 장면에 alt 텍스트 부여
- 타이틀/등록/실패 화면 전용 비주얼 분리

### STEP E

- `npm run build`
- 직접 URL 진입 검증
- 모바일 폭 검증
- 최소 두 엔딩 재현
- Story 01의 1회 플레이가 짧게 느껴지면 중간 사건과 귀환 전 대기 장면을 추가해 10분 이상으로 맞춘다.
- Story 02 착수 전에는 Story 01의 엔딩 리포트와 Story 02 시작점이 같은 세계관 문법을 공유하는지 확인한다.

## 8. 검증 기준

- `/story/1/intake` 진입 성공
- `/story/1/night-gate` 직접 URL 진입 성공
- `/story/1/ending-clear` 직접 URL 진입 성공
- 보급과 소음이 선택지에 따라 즉시 반영
- 조건 미달 선택지는 비활성화
- `ending-clear`, `ending-debt`, `ending-quarantine` 중 최소 두 개 이상 재현 가능
- 시작 화면에서 이름 입력과 임시 코드 발급 흐름이 확인 가능
- 실패 화면에서 검문/격리 문구가 행정적 형식으로 남는다

## 9. 다음 스토리로 넘길 것

- 현재 엔진은 Story 02부터 재사용한다.
- Story 02에서는 같은 상태값 2개를 유지하되, 외곽 보급 루트의 장거리 이동과 감시 리스크를 확장한다.
- Story 01의 소문과 조직은 후속 스토리 씨앗으로 재사용한다.
  - 북문 검문조 성향
  - 침상 브로커
  - 기록국 문구 수정
  - 외곽 수문 쪽 비공식 길잡이
- Story 02는 `외곽 보급 루트`, `통행 브로커`, `운반 부담`, `검문 예고`, `동행자 신뢰`를 중심으로 한 장거리 회수 이야기로 시작한다.
- Story 01의 엔딩은 Story 02의 진입 조건이기도 하다. `N-CLEAR`, `A-DEBT`, `Q-HOLD`는 단순 결과가 아니라 다음 스토리의 초기 상태값이다.
- 후속 문서에서는 `관리국가`, `병목 공간`, `아파트 주민 질서`를 더 깊게 쓰고, 영화적 영향은 문법만 가져오고 서사는 별도로 유지한다.
